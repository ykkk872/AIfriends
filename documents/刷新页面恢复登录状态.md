# 刷新页面时从云端恢复登录状态

> 对应 commit: `4b61d1285d332a7ea4238e0256e2806304c2fe59`

---

## 一、要解决的问题

用户登录后，登录信息（用户名、头像、access_token 等）存在前端的 Pinia store（内存）中。
浏览器一刷新，内存清空，用户看起来变成了"未登录"状态。
但浏览器 cookie 里的 `refresh_token` 还在（有效期 7 天），说明用户的登录会话并没有真正过期。

**目标：利用 cookie 中还存活的 refresh_token，在页面加载时自动恢复登录状态。**

---

## 二、实现思路

整体分三层：后端提供接口 → 前端页面加载时调用 → 前端路由和 UI 配合防闪烁。

### 第 1 步：后端新增"获取用户信息"接口

创建 `backend/web/views/user/account/get_user_info.py`：

```python
class GetUserInfoView(APIView):
    permission_classes = [IsAuthenticated]

    def get(self, request):
        user = request.user
        user_profile = UserProfile.objects.get(user=user)
        return Response({
            'result': 'success',
            'user_id': user.id,
            'username': user.username,
            'photo': user_profile.photo.url,
            'profile': user_profile.profile,
        })
```

在 `backend/web/urls.py` 中注册路由：

```python
path('api/user/account/get_user_info/', GetUserInfoView.as_view()),
```

这个接口要求必须带有效 token 才能访问（`IsAuthenticated`），返回当前用户的基本信息。

### 第 2 步：Pinia store 新增 `hasPulledUserInfo` 标志

在 `frontend/src/stores/user.js` 中新增：

```javascript
const hasPulledUserInfo = ref(false)

function setHasPulledUserInfo(newStatus) {
    hasPulledUserInfo.value = newStatus
}
```

这个布尔标志的含义：**是否已经向后端尝试过拉取用户信息**（不管成功还是失败）。
它的作用是让 UI 和路由守卫知道"异步请求还没回来，先别急着判断登录状态"，从而防止闪烁和误判。

### 第 3 步：App.vue 中页面加载时发请求

在 `frontend/src/App.vue` 的 `onMounted` 中：

```javascript
onMounted(async () => {
  try {
    const res = await api.get('/api/user/account/get_user_info/')
    const data = res.data
    if (data.result === 'success') {
      user.setUserInfo(data)        // 成功 → 恢复用户信息到 store
    }
  } catch (err) {
    // 失败（没登录或 token 全过期）→ 不做任何事，保持未登录
  } finally {
    user.setHasPulledUserInfo(true) // 无论成功失败，标记"已尝试过"

    // 如果当前页面需要登录但用户没登录 → 跳转登录页
    if (route.meta.needLogin && !user.isLogin()) {
      await router.replace({ name: 'user-account-login-index' })
    }
  }
})
```

请求的完整链路：
1. `api.get(...)` 发请求，自动带上 `access_token`（由 `api.js` 拦截器处理）
2. 如果 `access_token` 过期，`api.js` 的响应拦截器自动用 cookie 中的 `refresh_token` 刷新，然后重发
3. 如果 `refresh_token` 也过期了，请求失败，进入 catch

### 第 4 步：路由添加 `meta.needLogin` 标记 + 全局守卫

在 `frontend/src/router/index.js` 中，给每个路由添加 meta 信息：

```javascript
{ path: '/friend/', component: FriendIndex, meta: { needLogin: true } },
{ path: '/create/', component: CreateIndex, meta: { needLogin: true } },
{ path: '/',        component: HomepageIndex, meta: { needLogin: false } },
// ... 其他路由同理
```

添加全局前置守卫：

```javascript
router.beforeEach((to, from) => {
  const user = useUserStore()
  if (to.meta.needLogin && user.hasPulledUserInfo && !user.isLogin()) {
    return { name: 'user-account-login-index' }
  }
  return true
})
```

三个条件缺一不可：
- `to.meta.needLogin` — 目标页面声明需要登录
- `user.hasPulledUserInfo` — 已经尝试过拉取（避免请求还在路上就误判）
- `!user.isLogin()` — 确实没登录

### 第 5 步：导航栏防闪烁

在 `frontend/src/components/navbar/NavBar.vue` 中：

```html
<!-- 拉取完毕且未登录 → 显示登录按钮 -->
<RouterLink v-if="user.hasPulledUserInfo && !user.isLogin()" ...>登录</RouterLink>

<!-- 已登录 → 显示用户菜单 -->
<UserMenu v-else-if="user.isLogin()" />

<!-- 还在拉取中 → 两个条件都不满足，什么都不显示 -->
```

## 三、完整执行流程

```
用户打开 / 刷新页面
       │
       ▼
  App.vue onMounted() 自动触发
       │
       ▼
  GET /api/user/account/get_user_info/
  （自动带 access_token，过期则自动用 refresh_token 刷新后重试）
       │
       ├─ 成功 → store.setUserInfo(data) → 登录状态恢复
       │
       └─ 失败 → 保持未登录
       │
       ▼
  store.setHasPulledUserInfo(true)
       │
       ├─ 导航栏：根据 hasPulledUserInfo + isLogin() 显示正确内容
       └─ 路由守卫：开始正常拦截需要登录的页面
```

## 六、关键设计要点

| 设计点 | 为什么这样做 |
|---|---|
| 接口放在 `App.vue` 的 `onMounted` | App.vue 是根组件，打开/刷新页面时必定执行且只执行一次 |
| `hasPulledUserInfo` 标志 | 区分"没登录"和"还不知道有没有登录"，防止异步请求期间的 UI 闪烁和路由误判 |
| `finally` 块里设标志 | 确保无论请求成功、失败、网络超时，标志都能被设置 |
| 路由守卫三重条件 | `needLogin && hasPulledUserInfo && !isLogin()`，三个条件缺一不可 |
| 双重守卫 | App.vue 的 onMounted 处理首次加载的跳转；router.beforeEach 处理后续页面间跳转 |

## 七、复现时踩过的坑

### 坑 1：`route.needLogin` 写成了 `route.meta.needLogin`

**位置**：`App.vue` 的 `onMounted` → `finally` 块

```javascript
// ✗ 错误写法
if (route.needLogin && !user.isLogin()) { ... }

// ✓ 正确写法
if (route.meta.needLogin && !user.isLogin()) { ... }
```

**原因**：`needLogin` 是定义在路由的 `meta` 对象里的属性，不是路由对象本身的属性。`route.needLogin` 永远是 `undefined`，条件永远不成立，导致未登录用户访问需要登录的页面时不会被跳转。

---

### 坑 2：路由守卫中 `hasPulledUserInfo` 的两个错误

**位置**：`router/index.js` 的 `router.beforeEach`

```javascript
// ✗ 错误写法（两个问题）
if (to.meta.needLogin && !user.hasPulledUserInfo() && !user.isLogin()) { ... }

// ✓ 正确写法
if (to.meta.needLogin && user.hasPulledUserInfo && !user.isLogin()) { ... }
```

**问题 A — 加了 `()`**：`hasPulledUserInfo` 是 Pinia store 暴露的 `ref` 值（布尔值），不是函数，不能加 `()`。对比 `isLogin()` 是一个函数所以需要加 `()`，而 `hasPulledUserInfo` 是一个响应式变量，直接当属性读。

**问题 B — 多了 `!`**：`!user.hasPulledUserInfo` 的意思是"还没拉取过用户信息"，这会导致在请求还在路上时就拦截跳转——此时 `isLogin()` 也是 false（信息还没回来），用户会被立即踢到登录页，即使 token 其实是有效的。正确逻辑是 `user.hasPulledUserInfo`（已经拉取过了），只有确认"拉取完了且确实没登录"才应该拦截。

---

### 坑 3：NavBar 的 `v-if / v-else-if / v-else` 链导致闪烁

**位置**：`NavBar.vue` 导航栏右侧

```html
<!-- ✗ 错误写法：三个元素串在一条 v-if 链上 -->
<router-link v-if="user.isLogin()" ...>创作</router-link>
<router-link v-else-if="user.hasPulledUserInfo && !user.isLogin()" ...>登陆</router-link>
<UserMenu v-else />
```

**问题**：当页面刚加载、还在拉取用户信息时：
1. `user.isLogin()` → false → 跳过"创作"
2. `user.hasPulledUserInfo` → false → `v-else-if` 条件不满足 → 跳过"登陆"
3. 进入 `v-else` → **显示 `<UserMenu />`**

用户还没登录却闪了一下 UserMenu，这就是闪烁。

```html
<!-- ✓ 正确写法："创作"独立判断，"登陆"和 UserMenu 组成独立的 v-if 链 -->
<router-link v-if="user.isLogin()" ...>创作</router-link>

<router-link v-if="user.hasPulledUserInfo && !user.isLogin()" ...>登录</router-link>
<UserMenu v-else-if="user.isLogin()" />
```

**关键**："创作"按钮用独立的 `v-if`，不和后面的元素串链。"登录"按钮和 `UserMenu` 组成新的 `v-if / v-else-if` 链：
- 拉取中 → 两个条件都不满足 → **什么都不显示**（正确，避免闪烁）
- 拉取完毕 + 未登录 → 显示"登录"
- 已登录 → 显示 `UserMenu`

### 坑 4：登录报"系统异常，请稍后重试"

**位置**：后端 `login.py`

**现象**：用户名密码正确，但登录接口返回"系统异常，请稍后重试"。

**原因**：`authenticate()` 验证通过后，代码执行 `UserProfile.objects.get(user=user)` 查询用户的 Profile 记录。但数据库中该用户没有对应的 `UserProfile` 行，抛出 `UserProfile.DoesNotExist` 异常，被 `except:` 捕获后返回了笼统的错误信息。

**根本原因**：创建用户时没有同步创建 `UserProfile`（比如通过 `createsuperuser` 或直接操作数据库建的用户）。

**修复**：为该用户补建 Profile 记录即可。

**教训**：`except:` 裸异常捕获会吞掉所有错误信息，开发阶段建议改成：

```python
except Exception as e:
    print(f'Login error: {e}')  # 控制台能看到真实报错
    return Response({'result': '系统异常，请稍后重试'})
```

### 坑 5：`refresh_token.py` 中 `request.cookies`（小写）应为 `request.COOKIES`（大写）

**位置**：`backend/web/views/user/account/refresh_token.py`

```python
# ✗ 错误写法
refresh_token = request.cookies.get("refresh_token")

# ✓ 正确写法
refresh_token = request.COOKIES.get("refresh_token")
```

**原因**：Django 的 request 对象中，cookie 字典的属性名是大写的 `COOKIES`。小写 `cookies` 不存在，调用时抛 `AttributeError`，被 `except:` 吞掉后直接返回 401，导致 refresh token 刷新永远失败。

---

### 坑 6：`refresh_token.py` 中 `return` 漏写了 `response`

**位置**：`refresh_token.py` 旋转 refresh token 的分支

```python
# ✗ 错误写法
response = Response({...})
response.set_cookie(...)
return          # 返回了 None

# ✓ 正确写法
return response  # 返回构造好的响应
```

**原因**：前面花了好几行构造 `response` 对象并设置了新的 cookie，但最后 `return` 后面什么都没写，Python 返回 `None`，Django 拿到 `None` 就会报错，又被 `except:` 吞掉返回 401。

---

### 坑 7：`set_cookie` 中 `secure=True` 在 HTTP 开发环境下导致 cookie 丢失

**位置**：`login.py` 和 `refresh_token.py` 中的 `response.set_cookie(...)`

```python
# ✗ 开发环境的错误写法
response.set_cookie(
    key='refresh_token',
    value=str(refresh),
    httponly=True,
    samesite='Lax',
    secure=True,       # 只在 HTTPS 下生效
    max_age=86400 * 7,
)

# ✓ 开发环境的正确写法
response.set_cookie(
    key='refresh_token',
    value=str(refresh),
    httponly=True,
    samesite='Lax',
    secure=False,      # HTTP 下必须设为 False；生产环境（HTTPS）改回 True
    max_age=86400 * 7,
)
```

**原因**：`secure=True` 告诉浏览器"这个 cookie 只在 HTTPS 连接下才保存和发送"。本地开发用的是 `http://localhost`（HTTP），浏览器看到 `Secure` 标志后**根本不会保存这个 cookie**。所以登录虽然成功了，但 refresh_token cookie 从未被存下来，刷新页面后自然找不到。

**注意**：`login.py` 和 `refresh_token.py` 两个文件里都有 `set_cookie`，都要改。

### 坑 8：`localhost` 与 `127.0.0.1` 域名不一致导致 cookie 跨域丢失

**现象**：修完坑 5/6/7 后，重新登录再刷新，cookie 仍然没有带过来。

**原因**：前端运行在 `http://localhost:5173`，而 `api.js` 中配置的后端地址是 `http://127.0.0.1:8000`。对浏览器来说 `localhost` 和 `127.0.0.1` 是**两个不同的域**。登录时后端在 `127.0.0.1` 域下设了 cookie，但浏览器从 `localhost` 页面发跨域请求时，`SameSite=Lax` 策略下**不会带上属于 `127.0.0.1` 的 cookie**。

```javascript
// ✗ 域名不一致（前端 localhost，后端 127.0.0.1）
const BASE_URL = 'http://127.0.0.1:8000'

// ✓ 统一用 localhost
const BASE_URL = 'http://localhost:8000'
```

**修复后**：前端 `localhost:5173` → 请求 `localhost:8000`，域名一致，cookie 正常保存和发送。

**教训**：开发时前后端的域名必须保持一致（都用 `localhost` 或都用 `127.0.0.1`），否则浏览器会把它们当成不同的站点，cookie 无法跨域传递。
